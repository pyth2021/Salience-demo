Telemetry Detection Pipeline

The implemented pipeline connects the Cloudflare Pages website, beacon.js, Cloudflare Worker, Azure-hosted Flask /predict API, Cloudflare D1, and the Streamlit Dashboard.

Overview Live Stats Dashboard GitHub

High-Level Process Flow

The browser beacon collects only approved, minimized telemetry. The Cloudflare Worker validates the event, requests predictions from the Azure ML API, applies rule-based decision logic, and stores the complete assessment in Cloudflare D1.

1. Static Website A visitor interacts with the website hosted on Cloudflare Pages.
2. JavaScript Beacon beacon.js collects limited privacy-safe telemetry, including timing, interaction, session, browser, and coarse connection signals.
3. Cloudflare Worker Intake The Worker receives the telemetry event, validates the payload, filters unapproved fields, and prepares the model feature vector.
4. Rule-Based Signal Evaluation Worker rules evaluate obvious suspicious patterns such as rapid requests, scanner-like activity, unusual paths, and elevated error behaviour.
5. Worker to Azure ML API The Worker sends the approved feature vector to the Azure-hosted Flask /predict endpoint.
6. Dual-Model Prediction Gradient Boosting returns a traffic class and confidence, while Isolation Forest returns an anomaly signal and anomaly score.
7. Azure Returns Results The Azure API sends both model outputs back to the Cloudflare Worker.
8. Worker Decision Engine The Worker combines model predictions, anomaly signals, and rule-based indicators to calculate the final risk score, risk level, and recommended action.
9. Cloudflare D1 Storage D1 stores the minimized telemetry, model outputs, risk assessment, and recommendation for each processed event.
10. Streamlit Dashboard The dashboard presents traffic classifications, anomaly results, risk levels, recommendations, and evaluation evidence.
Privacy-safe telemetry collection Worker ↔ Azure prediction round trip Dual-model detection Rule-based decision engine Cloudflare D1 event storage Streamlit reporting dashboard

Worker and Azure Communication

The Worker and Azure API communicate in both directions. The Worker sends the feature vector for prediction, and Azure returns the supervised and unsupervised model results.

Worker to Azure

  • Validated feature vector
  • Approved 14-feature model schema
  • No raw IP address
  • No exact location
  • No persistent browser identifier

Azure to Worker

  • Gradient Boosting traffic class
  • Supervised prediction confidence
  • Isolation Forest anomaly result
  • Isolation score
  • Model metadata used by the decision engine

Detection Methods

The prototype combines deterministic Worker rules with two trained machine-learning models. Each method contributes a different signal to the final decision.

Method Learning Type Purpose Output Implementation
Rule-Based Detector Deterministic rules Identifies clear suspicious patterns, including rapid requests, suspicious page categories, high error behaviour, and scanner-like activity. Rule indicators and score contribution Cloudflare Worker logic
Isolation Forest Unsupervised anomaly detection Determines whether an event differs from the normal benign traffic patterns used during training. Normal or anomalous signal plus anomaly score isolation_forest_model.pkl
Gradient Boosting Supervised multi-class classification Classifies telemetry as human, good bot, bad bot, or scanner using labelled synthetic training examples. Traffic class plus prediction confidence gradient_boosting_model.pkl

Decision Engine

The system does not automatically block visitors. It produces an advisory risk assessment and recommended action for dashboard review.

Example Condition Risk Level Recommended Action
Gradient Boosting predicts human traffic, Isolation Forest reports normal behaviour, and no strong rule indicators are present. Low Allow
The models disagree, confidence is limited, or some suspicious rule indicators are present. Medium Monitor
Gradient Boosting predicts bad bot or scanner activity, the event is anomalous, or multiple strong suspicious indicators are present. High Flag for review

These examples summarize the decision logic conceptually. The final score and recommendation are generated consistently by the Worker using the implemented rule thresholds and returned model outputs.

Stored Event Record

After the decision engine completes its assessment, the Worker stores the complete result in Cloudflare D1 for later analysis.

Stored Telemetry

  • Approved minimized feature values
  • Page and interaction categories
  • Request and session behaviour
  • Coarse user-agent and TLS-derived properties
  • Event timestamp and processing metadata

Stored Detection Results

  • Gradient Boosting prediction
  • Prediction confidence
  • Isolation Forest result
  • Isolation score
  • Final risk score
  • Final risk level
  • Recommended action

Project Boundaries

In Scope

  • Professional static website and JavaScript beacon
  • Cloudflare Worker telemetry intake endpoint
  • Privacy validation and minimized telemetry schema
  • Synthetic dataset and feature-engineering pipeline
  • Rule-based detector
  • Isolation Forest anomaly detection
  • Gradient Boosting traffic classification
  • Azure-hosted Flask prediction API
  • Cloudflare D1 storage

Out of Scope

  • Real sponsor production traffic or personal data
  • Production-grade blocking or CAPTCHA enforcement
  • Automated rate limiting or active mitigation

  • Invasive behavioural tracking
  • Persistent browser fingerprinting
  • Formal end-to-end latency benchmarking
  • Commercial-ready security product deployment
